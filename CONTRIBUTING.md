# Contributing

This project contains two coupled runtimes:

- a Python MCP server in `python-server/`;
- an ArcGIS Pro Add-In in `arcgis-addin/`.

Keep changes small, testable, and aligned with the existing module structure.

## Local checks

Run these checks before opening a pull request:

```powershell
.\scripts\validate_release.ps1
```

Or run the individual checks:

```powershell
cd python-server
python -m pip install -r requirements-dev.txt
python -m ruff check . --no-cache
python -m ruff format . --check --no-cache
python -m pytest -q
python -m unittest discover -s tests -v
```

```powershell
cd ..\arcgis-addin
dotnet build .\ArcGisMcpAddin.sln
```

With ArcGIS Pro open and the Add-In loaded:

```powershell
cd ..\python-server
python test_connection.py
```

## Tool contract

- Add new MCP tools in `python-server/arcgis_mcp_server.py`.
- Place tool implementation in the matching `python-server/tools/` module.
- Add Add-In commands to `CommandNames.All` only when C# handles the command.
- Add Python-only wrappers to `ToolNames.All` when they call existing Add-In commands.
- Keep `ping` as an internal Add-In command, not a public MCP tool.
- Keep `spatial_join` as a single public MCP tool.

Update `python-server/tests/test_operational_contracts.py` when the public tool
contract changes intentionally.

## Repository hygiene

Do not commit:

- local project files or personal paths;
- build output from `bin/` or `obj/`;
- Python cache directories;
- `.esriAddinX` packages generated by MSBuild;
- temporary validation output.

Run `git diff --cached --check` before committing.
